# 工作区串行过渡决策（2026-03-28）

## 1. 决策摘要

从 `2026-03-28` 开始，Spotlight 服务端不再把所有活跃任务都视为“全局只能有一个”。

当前过渡规则调整为：

- 不同工作区的任务可以并行推进
- 同一工作区的任务仍然保持串行

这是一个刻意的中间态，不是最终态。

## 2. 为什么现在先这么做

本次调整的背景是：

- 现有实现已经支持多项目、多 Agent 的产品目标表达
- 但执行层实际上把 `Claimed` / `Running` 任务做成了全局串行
- 同时，当前 Git 执行上下文仍然绑定项目主工作区，而不是独立 worktree

如果直接放开“同一工作区多任务并行”，会立即撞上这几个风险：

- 共享工作区互相污染
- 自动分支切换互相干扰
- 恢复逻辑仍然绑定 `task + thread + 主工作区`
- 出错后无法稳定回收现场

所以本次先只解除“跨工作区也被全局锁死”的限制。

## 3. 本次实现范围

本次代码调整包含：

- 活跃任务冲突判定改为按工作区判断
- auto-claim 在选择任务时会跳过已被同工作区活跃任务占用的队列
- watchdog 的并行回收逻辑只回收同工作区冲突任务
- 补了对应回归测试

本次明确不做：

- 同一工作区并行执行
- execution slot 正式建模
- worktree / clone 隔离工作区
- agent mailbox / dependency graph

## 4. 临时边界

在 `execution slot + isolated workspace` 落地前，以下规则继续成立：

- 同一项目主工作区只允许一个活跃任务
- 不允许把“跨工作区并行”误解为“同仓库并行安全”
- 如果多个项目指向同一个物理工作区，仍应视为潜在冲突源

换句话说，本次是“缩小串行范围”，不是“完成并行架构”。

## 5. 后续实施顺序

建议后续按这个顺序推进：

1. 把 `execution slot` 和 `workspace lease` 引入运行模型
2. 把 Git 执行从共享主工作区切换到独立 worktree
3. 补任务依赖与 handoff 原语
4. 再讨论同一工作区内的受控并行
