# 0.1.0 编译与重启动作说明

## 背景

Spotlight 当前是任务驱动平台，而不是直接绕过任务流的“一键运维脚本”。

因此，项目视图中的“本地编译重启”和“云端安装重启”在 `0.1.0` 的最小落地，应该先体现为两类可审计、可认领、可回放的预制任务。

这样做的目的有三点：

- 保持所有动作都进入统一任务流
- 让 Agent 在执行前先分析项目技术栈和风险
- 把部署和重启知识沉淀到项目内文档，例如 `deploy.md`

## 0.1.0 的最小落地

在项目视图中增加两个按钮：

- `本地编译重启`
- `云端安装重启`

点击后，不直接脱离任务系统执行，而是分别创建对应的预制任务。

## 本地编译重启

点击 `本地编译重启` 后，系统应创建一条任务，至少包含以下要求：

1. 识别当前项目类型与主要语言
2. 优先判断 Rust、C++、Python、JavaScript / TypeScript，也允许补充其他技术栈
3. 判断依赖安装、构建、打包、启动、重启入口是否存在
4. 能执行时完成一次本地编译/打包和服务重启
5. 不能执行时明确缺失项、风险和人工确认项
6. 新增或更新项目根目录下的 `deploy.md`

系统可以基于目录中的常见构建清单给出“初步识别”，例如：

- `Cargo.toml`
- `package.json`
- `pyproject.toml`
- `CMakeLists.txt`

但该识别只是提示，不应替代 Agent 的实际分析。

## 云端安装重启

点击 `云端安装重启` 后，系统应要求补充最小远端信息，例如：

- 服务器 IP 或域名
- SSH 用户名
- SSH 端口
- 认证方式
- 凭据说明
- 部署目录
- 服务名或重启命令

随后创建一条云端部署任务，至少包含以下要求：

1. 结合本地项目内容识别技术栈与构建方式
2. 规划远端依赖安装、构建、发布、重启与校验步骤
3. 条件具备时通过 SSH 执行；条件不足时明确阻塞项
4. 新增或更新 `deploy.md`
5. 对覆盖发布、系统级安装、服务重启等危险动作补充风险与回滚说明

## 安全约束

这两个动作都需要遵守以下约束：

- 不要假设当前目录一定是完整代码仓库
- 遇到文档、Office、图片、压缩包或其他二进制内容时不能臆造
- 不要对项目外目录做破坏性修改
- 不要把明文密码写入任务描述、`deploy.md` 或长期日志
- 云端优先使用 SSH 证书、私钥路径或凭据别名

如果需要密码登录，建议在真正启动任务前，由操作者临时补充，而不是长期保存在任务模板里。

## 后续扩展

后续版本可以继续扩展：

- 把项目级部署参数持久化成结构化配置，而不是每次弹窗输入
- 支持多环境，例如测试、预发、生产
- 将 `deploy.md` 与结构化部署配置互相同步
- 为远端发布增加审批、危险操作提示和回滚保护
- 基于项目技术栈自动推荐更细的执行模板
