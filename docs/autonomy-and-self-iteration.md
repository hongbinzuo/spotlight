# 自治演进与多租户自迭代架构

## 0. 核心问题

Spotlight 要回答的不是"能不能自动执行任务"，而是：

1. 多人多项目并行时，Agent 能否**自治**（不靠人盯着也能持续推进）？
2. 平台自身能否**自迭代**（用自己的任务系统改进自己）？
3. 不同用户能否**选择演进节奏**（有人要稳定版，有人要最新版）？

## 1. 四个自治层次

### 1.1 自动（Automation）—— 已实现

当前状态：
- auto-mode Agent 自动认领、执行、恢复
- watchdog 自动回收卡住任务
- 完成后自动生成后续任务和待回答问题
- 服务重启时自动归一化状态

缺口：
- 认领前没有重评估门禁（刚加的 reassess 还未接入自动循环）
- 没有跨项目的全局调度

### 1.2 自治（Autonomy）—— 部分实现

当前状态：
- Agent 有决策规则文档 (`agent-autonomy-and-decision-rules.md`)
- Agent 可以自行创建后续任务
- 项目约束通过记忆层持久化

缺口：
- Agent 不会主动发现"我做的事已经有人做了"
- Agent 不会主动评估"当前项目最需要做什么"
- 没有"项目级策略 Agent"——一个不执行任务但持续评估全局状态的角色

### 1.3 自学习（Self-Learning）—— 骨架存在

当前状态：
- 版本化事实记忆层可以沉淀约束和摘要
- 任务完成报告会写入记忆

缺口：
- 没有"模式识别"：哪些类型的任务总是失败？哪些提示词模板效果好？
- 没有"经验回放"：新 Agent 接手时不会去看历史成功任务的执行模式
- 没有反馈闭环：用户对任务结果的满意度没有回收

### 1.4 自迭代（Self-Iteration）—— 设计阶段

Spotlight 自举的终极形态：平台用自己的任务系统来改进自己。

## 2. 多项目多人并行的自治架构

### 2.1 当前限制

```
用户 A ──┐
用户 B ──┼── 单实例 Server ── 串行执行（同时只有 1 个 Running 任务）
Agent 1 ─┤
Agent 2 ─┘
```

这在单人开发 Spotlight 自身时够用，但无法支持：
- 用户 A 的项目 P1 和用户 B 的项目 P2 同时推进
- 同一项目内多个 Agent 并行执行不同任务

### 2.2 目标架构

```
┌─────────────────────────────────────────────┐
│              中心协调服务                      │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐  │
│  │ 任务调度 │  │ 记忆/事实 │  │ 策略 Agent │  │
│  └─────────┘  └──────────┘  └────────────┘  │
└──────┬──────────┬──────────────┬─────────────┘
       │          │              │
  ┌────┴───┐ ┌───┴────┐  ┌─────┴─────┐
  │ 执行槽1 │ │ 执行槽2 │  │ 执行槽 N  │
  │ Agent-A │ │ Agent-B │  │ Agent-C   │
  │ 项目 P1 │ │ 项目 P2 │  │ 项目 P1   │
  │ 工作区1 │ │ 工作区2 │  │ 工作区1'  │
  └────────┘ └────────┘  └───────────┘
```

关键变化：

| 维度 | 现在 | 目标 |
|------|------|------|
| 执行并行度 | 全局串行（1 个 Running） | 按项目并行、项目内可配串行/并行 |
| 调度 | Agent 自己抢 | 中心调度 + 重评估门禁 |
| 工作区隔离 | 共享同一工作目录 | 每个执行槽独立 worktree 或 clone |
| 冲突检测 | 只检查任务状态 | 检查代码库 + 任务状态 + git 分支 |

### 2.3 策略 Agent（新角色）

不执行任务，只做三件事：

1. **全局评估**：定期扫描所有项目的任务状态，标记需要重评估的任务
2. **优先级调整**：根据项目进度、阻塞关系、用户反馈动态调整任务优先级
3. **冲突预防**：在 Agent 认领任务前，检查是否有重叠工作正在进行

```
每 N 秒:
  for project in all_projects:
    tasks = project.tasks.filter(needs_decision)
    for task in tasks:
      evidence = check_codebase(project.workspace)
      siblings = project.tasks.filter(done_or_running)
      decision = reassess(task, evidence, siblings)
      apply(decision)  // DONE / REOPEN / RESTART / CANCEL
```

## 3. 自学习机制

### 3.1 经验沉淀

每个任务完成后，自动提取并存入记忆层：

```
memory_item:
  scope: project
  kind: execution_pattern
  content:
    task_type: "编译重启"
    success_rate: 0.8
    avg_duration: 15min
    common_failures: ["依赖缺失", "端口占用"]
    effective_prompt_patterns: ["先识别技术栈再动手"]
    anti_patterns: ["不检查就直接 npm install"]
```

### 3.2 模式识别

Agent 在认领新任务时，查询记忆层的历史模式：

```
我要做"本地编译重启"类型的任务。
历史数据显示：
- 这类任务 80% 成功
- 常见失败原因是"依赖缺失"
- 有效策略是"先跑 cargo check 确认编译环境"
→ 我应该先检查环境再动手
```

### 3.3 反馈回收

```
任务完成 → 用户/验收者评价（满意/不满意/需要返工）
                     ↓
             写入 memory_item
                     ↓
        下次同类任务参考历史评价
```

## 4. 自迭代：通过 GitHub Repo 完成进化

### 4.1 核心机制

Spotlight 的自迭代不是魔法，而是一个标准的 CI 驱动演进循环：

```
┌─────────────────────────────────────────────┐
│                  自迭代循环                    │
│                                              │
│  1. 策略 Agent 发现缺口/改进点                 │
│        ↓                                     │
│  2. 创建改进任务（到 Spotlight 自身项目）        │
│        ↓                                     │
│  3. 执行 Agent 认领、实现、提交到 task 分支     │
│        ↓                                     │
│  4. CI 跑测试、lint、构建                      │
│        ↓                                     │
│  5. 合并到 main（自动或人工审批）               │
│        ↓                                     │
│  6. 新版本发布到 GitHub Releases               │
│        ↓                                     │
│  7. 各用户/租户按自己的策略决定是否升级          │
│        ↓                                     │
│  回到 1                                       │
└─────────────────────────────────────────────┘
```

### 4.2 版本策略与用户选择

用户（或租户）可以选择三种模式：

| 模式 | 行为 | 适合谁 |
|------|------|--------|
| **锁定版本（GA Pin）** | 锁定到某个 release tag，不自动升级 | 生产环境、保守用户 |
| **跟随稳定（Follow Stable）** | 自动跟随 `stable` 分支，只接收已验证的更新 | 日常使用 |
| **自迭代（Self-Evolve）** | 允许本地 Agent 提交改进，参与上游贡献 | 开发者、早期用户 |

```
GitHub Repo
├── main            ← 最新开发版
├── stable          ← 经过 CI + 人工验证的稳定版
├── releases/
│   ├── v0.1.0      ← GA 锁定点
│   ├── v0.1.1
│   ├── v0.2.0      ← 下一个 GA 锁定点
│   └── ...
└── contributions/
    ├── agent-A-improve-prompt/    ← Agent 提交的改进 PR
    ├── agent-B-fix-recovery/
    └── ...
```

### 4.3 自迭代的安全边界

Agent 可以自行决定的：
- 补测试、修 bug、优化提示词模板
- 改进内部代码结构（重构）
- 增强已有功能（在不改变接口的前提下）

Agent 必须请示的：
- 改变数据模型（迁移）
- 改变 API 接口（破坏兼容性）
- 改变安全/权限边界
- 改变计费/租户模型
- 删除功能

### 4.4 跨用户的学习共享

```
用户 A 的 Agent 发现：
  "编译重启任务在 Rust 项目上，先跑 cargo check 成功率从 60% 提升到 90%"
        ↓
  写入本地记忆层
        ↓
  (可选) 提交为 PR 到上游 repo:
  "改进 build_local_build_restart_task 提示词，增加 cargo check 预检步骤"
        ↓
  上游合并后，所有 follow-stable 的用户自动获得这个改进
```

这就是"通过 GitHub repo 完成进化"的具体含义——不是 Agent 直接改别人的环境，而是通过 PR → Review → Merge → Release 的标准流程，把个体学习变成集体进化。

## 5. 实现路径

### 阶段 1（当前 → 0.1.x 末期）

- [x] 基础自动化：auto-mode、watchdog、任务恢复
- [x] 重评估门禁：reassess API + 提示词模板
- [x] 版本化事实记忆层骨架
- [ ] **接入自动循环**：auto-mode 认领前调 reassess
- [ ] **经验沉淀**：任务完成后自动写入执行模式记忆

### 阶段 2（0.2.x）

- [ ] **策略 Agent**：独立角色，持续评估全局状态
- [ ] **按项目并行执行**：每个项目独立执行槽
- [ ] **工作区隔离**：git worktree 或独立 clone
- [ ] **模式识别**：查询历史执行模式辅助决策

### 阶段 3（0.3.x）

- [ ] **自迭代循环**：Agent 对 Spotlight 自身提交改进
- [ ] **CI 门禁**：改进必须通过测试才能合并
- [ ] **版本策略**：GA Pin / Follow Stable / Self-Evolve
- [ ] **跨用户学习共享**：通过 PR 机制分享执行模式改进

### 阶段 4（0.4.x+）

- [ ] **多租户隔离**：不同团队/组织独立运行
- [ ] **反馈回收闭环**：用户满意度驱动提示词和策略优化
- [ ] **联邦学习**：多实例共享脱敏后的执行模式（可选）
