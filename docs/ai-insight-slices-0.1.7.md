# 0.1.7 AI 洞察与管理能力规划

## 1. 目标

`0.1.7` 的目标不是一次性做完完整 AI 平台，而是把 Spotlight 的 AI 控制面和高价值洞察场景收敛成第一版可上线的产品切片：

- 管理员可以配置 Provider、预算、场景开关和默认模型路由
- 项目可以稳定产出日报总结、构建失败解释、验收辅助、工期预测和 Token 节省建议
- 洞察结果具备缓存、审计、预算约束和可回看历史
- 洞察结论明确标注为“分析结果”，不冒充事实来源

本版本重点是“低成本、可治理、可追踪”的管理 AI，不是让 AI 直接替代执行 Agent。

## 2. 范围边界

### 2.1 本版本纳入

- AI Provider 与模型包配置入口
- 组织级、项目级的洞察预算与默认路由
- 内建洞察场景注册表
- 洞察任务触发方式：`manual`、`scheduled`、`event_driven`
- 洞察结果缓存、运行历史、失败重试与审计记录
- 五类首发场景：日报总结、构建失败解释、验收辅助、工期预测、Token 节省分析

### 2.2 本版本暂不纳入

- 自定义 Prompt 编排工作流平台
- 多阶段洞察审批流
- 任意上传外部文档后做通用知识库问答
- 重度 BI 报表与完整财务预测系统
- 需要长上下文原始日志直喂大模型的昂贵分析链路

说明：

- `0.1.7` 首先解决“项目管理和交付判断”问题，不追求全场景 AI 化
- 复杂预算、计费权益、企业分层由 `0.1.8` 继续细化

## 3. 首发洞察场景

### 3.1 日报总结 `daily_summary`

目标：

- 为项目成员、负责人、管理者生成统一口径的日摘要

输入契约：

- 任务状态增量
- 新增与关闭任务数
- 构建与部署结果摘要
- 风险事件与阻塞项
- 里程碑变化

输出契约：

- 今日进展摘要
- 当前阻塞项
- 明日建议动作
- 是否需要升级关注

触发方式：

- 默认 `scheduled`
- 支持项目管理员手动重跑

低 Token 策略：

- 先按任务、构建、风险做结构化聚合
- 默认只发送过去 24 小时增量，不重扫全历史

### 3.2 构建失败解释 `build_failure_explainer`

目标：

- 帮助团队快速理解一次失败更像“环境问题、依赖问题、代码问题还是配置问题”

输入契约：

- 标准化失败签名
- Top-K 错误摘要
- 相关仓库、分支、任务、最近变更元数据
- 最近成功构建对照信息

输出契约：

- 失败归因候选
- 影响范围
- 建议排查顺序
- 是否值得自动生成后续任务

触发方式：

- 默认 `event_driven`

低 Token 策略：

- 不上传整份原始日志
- 仅发送错误摘要、堆栈片段和结构化元数据

### 3.3 验收辅助 `acceptance_assistant`

目标：

- 帮助验收人更快判断“该看什么、风险在哪、哪些证据还不够”

输入契约：

- 任务描述
- 任务运行总结
- 变更文件摘要
- 测试结果摘要
- 风险与危险操作摘要
- 验收产物链接

输出契约：

- 验收检查清单
- 可能遗漏的验证点
- 建议追问问题
- 建议人工重点抽查区域

触发方式：

- 默认在进入 `pending_acceptance` 时 `event_driven`
- 支持验收人手动重跑

低 Token 策略：

- 优先消费 `acceptance-and-artifacts.md` 定义的结构化产物
- 不直接拼接完整 diff 和长会话全文

### 3.4 工期预测 `delivery_eta_forecast`

目标：

- 结合最近交付速度和阻塞状态，给出“按当前趋势大概率何时完成”的分析区间

输入契约：

- 未完成任务规模与类型分布
- 最近 7/14/30 天完成速度
- 失败与返工率
- 验收等待时间
- 人员负载与关键路径阻塞
- 构建稳定性与回滚频率

输出契约：

- 预计完成时间区间
- 置信度
- 拖期主因
- 建议采取的压缩工期动作

触发方式：

- 默认 `scheduled`
- 支持里程碑前手动运行

边界说明：

- 输出必须明确为“预测区间”而不是承诺日期
- 当样本不足或数据质量过低时，应退化为“无法可靠预测”

### 3.5 Token 节省分析 `token_efficiency_advisor`

目标：

- 用更低成本维持洞察质量，优先发现可以缓存、降级模型、缩短输入的场景

输入契约：

- 各场景 token 消耗
- 缓存命中率
- 升级到大模型的频率
- 相似输入重复率
- 洞察失败率与人工否定率

输出契约：

- 建议降级或升级的模型路由
- 建议调整的缓存 TTL
- 建议新增的结构化预摘要字段
- 预计节省的 token 比例与风险提示

触发方式：

- 默认 `scheduled`
- 支持管理员 `manual` 运行

边界说明：

- 该场景服务于“洞察系统自身优化”，不直接给业务成员展示复杂 token 明细

## 4. AI 控制面最小产品形态

### 4.1 Provider 管理

- 配置多个 Provider、Endpoint、密钥引用和可用模型
- 支持组织默认 Provider 与项目级覆盖
- 支持连接测试与能力标记

### 4.2 场景配置

- 场景开关
- 默认模型等级
- 触发方式
- 输入窗口大小
- 单次 token 上限
- 缓存 TTL
- 升级模型策略

### 4.3 预算与治理

- 组织级月预算
- 项目级预算配额
- 场景级 token ceiling
- 超预算后的降级、排队或停止策略

### 4.4 历史与审计

- 洞察运行历史
- 命中缓存与未命中原因
- Provider / Model / Scenario 元数据
- 输入时间窗和 supporting signals
- 失败原因与重试记录

## 5. 数据与接口建议

建议在现有占位基础上补齐以下一等对象：

- `ai_scenarios`
- `ai_scenario_bindings`
- `ai_insight_runs`
- `ai_insight_cache`
- `ai_budgets`

建议最小接口：

- `GET /admin/ai/scenarios`
- `PATCH /admin/ai/scenarios/:key`
- `GET /admin/ai/insight-runs`
- `POST /projects/:project_id/insights/:scenario_key/run`
- `GET /projects/:project_id/insights/:scenario_key/history`

关键约束：

- 执行 Agent 的 Provider 配置与管理 AI 的 Provider 配置相关但分离
- 所有洞察结果必须记录模型、时间窗、置信度与 supporting signals
- 预测和解释类输出必须标记为“分析结果，不是源事实”

## 6. 推荐实施顺序

建议按以下顺序推进：

1. `daily_summary`
2. `build_failure_explainer`
3. `acceptance_assistant`
4. `delivery_eta_forecast`
5. `token_efficiency_advisor`

排序理由：

- 前三项最容易直接挂接已有任务、构建、验收数据
- 工期预测价值高，但依赖前面几类结构化信号更稳定
- Token 节省分析要在已有洞察运行样本出现后才真正有价值

## 7. 测试、性能与安全要求

### 7.1 必需测试

- 场景路由单元测试
- Provider 选择与失败回退集成测试
- 重复输入缓存命中回归测试
- 验收辅助输出契约测试
- 工期预测“样本不足退化”回归测试
- Token 分析预算阈值测试

### 7.2 安全关注点

- Provider 密钥必须引用安全存储，而不是明文回显
- 组织与项目的洞察结果必须严格隔离
- 后台只有具备权限的角色可查看预测和预算明细
- 洞察运行和配置变更都必须写入审计

### 7.3 性能关注点

- 日报和失败解释默认走增量窗口
- 高频事件驱动场景必须支持去重与限流
- 缓存命中率、平均 token 消耗、平均分析延迟必须可观测

## 8. 风险与注意事项

- 不要把 AI 洞察做成原始日志大吞吐系统，否则成本和延迟会失控
- 不要把工期预测包装成承诺日期，否则会误伤管理决策
- 不要让验收辅助替代人工责任链，它只能帮助聚焦，不负责拍板
- 不要在 `0.1.7` 提前做重型 Prompt 工作流编排，先把场景、预算和缓存边界做稳
