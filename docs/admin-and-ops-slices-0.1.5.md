# 0.1.5 后台与运维控制台切片规划

## 1. 目标

`0.1.5` 的目标不是一次性做完完整后台，而是先交付一版可用的后台控制面：

- 管理员无需桌面端即可查看核心配置和运行状态
- 项目配置、人员管理、运行态、风险态、监控态有独立入口
- 为 `0.1.7` 的 AI 控制面、`0.1.8` 的计费主体与权益、`0.1.9` 的安全审计增强预留稳定后台骨架

本次切片重点是“后台面板与管理入口”，不是提前实现完整企业计费系统或完整合规中心。

## 2. 范围边界

### 本版本纳入

- 后台 Web 壳与基础导航
- 项目配置
- 人员与角色/能力管理
- Agent 与 Runtime 状态
- 系统监控面板
- 风险面板第一版
- 计费与部署配置视图

### 本版本暂不纳入

- 完整订阅、账单生成、发票结算链路
- 完整 AI Provider 管理与预算策略中心
- 复杂审批编排、审计导出工作流
- 多租户私有化交付工具链本体

说明：

- `0.1.5` 的计费相关工作只做“后台入口、配置视图、状态透出、部署模式感知”，不提前吞并 `0.1.8` 的计费主体与权益实现。
- `0.1.5` 的风险面板只做第一版聚合视图与基础操作，不提前做完整回滚与合规闭环。

## 3. 切片拆分

### 3.1 `[0.1.5] 后台 Web 壳`

目标：

- 建立独立于桌面端的后台入口、布局、导航、权限骨架和基础会话能力

范围：

- 后台首页框架
- 顶部/侧边导航
- 基础登录态占位或管理员身份占位
- 后台页面路由与空态
- 通用列表页、详情页、筛选条骨架

最小验收：

- 可以进入后台首页
- 可以在不同后台页面之间切换
- 页面加载失败时有统一错误态

必需测试：

- 后台加载冒烟测试
- 未授权访问回归测试

### 3.2 `[0.1.5] 项目配置`

目标：

- 让管理员在后台查看和维护项目级配置，而不是依赖桌面端或手工改文件

范围：

- 项目基本信息
- 工作区根目录配置
- 项目可见性与成员可访问范围
- 默认工作流/验收人/审批人占位配置
- 允许的 Runtime Target 与 Provider 策略入口

依赖：

- 项目模型
- 可见性规则
- 工作流默认配置模型

必需测试：

- 项目可见性测试
- 项目配置 API 集成测试
- 配置变更回归测试

### 3.3 `[0.1.5] 人员与能力管理`

目标：

- 提供后台的人、团队、角色、能力授权入口

范围：

- 用户目录
- 团队目录
- 角色分配
- 能力授予与撤销
- 项目成员关系
- 停用/冻结入口

边界说明：

- 任务标题沿用“能力管理”表述，以符合现有文档中“管理 capability，而不只管理 title”的原则。
- 页面展示可以同时体现角色与能力，但数据抽象应优先能力边界。

必需测试：

- 权限边界测试
- 项目成员可见性测试
- 角色/能力变更回归测试

### 3.4 `[0.1.5] Agent 与 Runtime 状态`

目标：

- 让运维和管理员能看到执行面当前是否健康、是否可调度

范围：

- Agent 注册表
- 在线/离线状态
- 自动模式状态
- 当前执行任务
- Runtime Target 基础健康态
- 隔离、禁用、摘除入口占位

必需测试：

- Agent 状态 API 集成测试
- Runtime 状态聚合测试
- 运行态刷新回归测试

### 3.5 `[0.1.5] 系统监控面板`

目标：

- 补齐“运行对象状态”和“系统级监控”之间的缺口，形成后台总览

范围：

- 服务健康总览
- 队列深度
- 失败任务趋势
- 重试峰值
- 会话断连数
- 通知投递健康度
- 存储与保留策略健康度占位

边界说明：

- 该任务聚焦聚合视图和核心指标接口，不要求一次性接入完整 observability 平台。

必需测试：

- 监控聚合接口集成测试
- 列表/看板性能检查
- 指标空数据回归测试

### 3.6 `[0.1.5] 审计和风险中心第一版`

目标：

- 交付一版可用的风险面板，让管理员能集中查看高风险事件和关键审计记录

范围：

- 危险操作列表
- 审计检索入口
- 审批/验收历史聚合
- 策略判定历史占位
- 风险级别筛选
- 基础导出占位或下载入口占位

边界说明：

- 保留现有任务标题，风险面板作为该任务的首要落点。
- 真正的回滚安全闭环与更强审计链路仍归属 `0.1.9`。

必需测试：

- 危险操作可见性测试
- 审计查询 API 集成测试
- 未授权访问回归测试

### 3.7 `[0.1.5] 计费与部署配置视图`

目标：

- 在后台提前落下计费与部署模式的管理入口，避免后续 `0.1.8` 没有承载界面

范围：

- 当前部署模式展示
- 套餐/权益状态占位展示
- 计量摘要占位展示
- 私有化部署元数据展示
- 账单与订阅入口占位
- 后台对部署模式差异的 UI 分支

边界说明：

- 本任务只建设配置与展示视图，不要求完成完整计费域模型。

必需测试：

- 部署模式分支回归测试
- 计费配置视图集成测试
- 不同部署模式可见性测试

## 4. 推荐实施顺序

建议按以下顺序推进：

1. `[0.1.5] 后台 Web 壳`
2. `[0.1.5] 项目配置`
3. `[0.1.5] 人员与能力管理`
4. `[0.1.5] Agent 与 Runtime 状态`
5. `[0.1.5] 系统监控面板`
6. `[0.1.5] 审计和风险中心第一版`
7. `[0.1.5] 计费与部署配置视图`

排序理由：

- 先把后台容器和导航打稳，再接配置与权限边界
- 运行态和监控态要共享汇总接口，适合连续实现
- 风险与计费视图依赖更稳定的后台骨架和部署模式信息

## 5. 与现有版本规划的关系

- `0.1.5`：建立后台控制面的可用骨架与核心入口
- `0.1.7`：在后台接入 AI 控制面和洞察配置
- `0.1.8`：补齐计费主体、权益、部署模式的真实业务能力
- `0.1.9`：补齐更强的审计、危险操作识别和回滚安全闭环

## 6. 风险与注意事项

- 不要把“人员管理”做成只认头衔、不认能力的弱权限模型
- 不要把“Agent 状态页”误当成“系统监控面板”，两者粒度不同
- 不要把 `0.1.5` 的计费配置视图扩张成完整账单系统
- 不要让后台依赖桌面端本地状态或 Provider 原生事件格式
- 任何管理操作都需要可审计，哪怕先是占位链路
