# AGENTS.md

本文件是 `C:\Users\zuoho\code\spotlight` 内所有人类协作者、Agent、以及不同会话共享的执行手册。

Spotlight 的目标不只是“把产品做出来”。
更重要的是：用 Spotlight 自己的任务协作方式，把 Spotlight 一步一步自举出来。

## 0. 语言规则

本仓库从现在开始统一采用中文为主。

硬性规则：

- 对话沟通默认使用中文
- 新增文档默认使用中文
- 任务标题和任务描述默认使用中文
- 注释、说明、提交说明、测试命名，能用中文表达时优先中文

允许例外：

- API、协议字段、代码标识符、第三方库名称可保留英文
- 引用官方文档、标准名词、命令行参数时可保留英文原文
- 必须和上游系统兼容的文件格式，不强行翻译字段名

## 1. 工作原则

### 1.1 唯一事实来源

产品和架构决策的事实来源是：

- `docs/`
- 尤其是：
  - `docs/product-constraints-v1.md`
  - `docs/system-architecture.md`
  - `docs/data-model-v2.md`
  - `docs/platform-abstractions-v1.md`
  - `docs/admin-and-ops-console.md`
  - `docs/ai-management-and-insight-engine.md`

如果代码和文档冲突：

- 不能悄悄忽略
- 要么把代码对齐文档
- 要么在同一次修改里同步更新文档

### 1.2 产品定位

Spotlight 不是：

- 某一个 CLI 的壳子
- 只有任务列表和聊天框的工具

Spotlight 要成长为：

- 多人协作的 AI 执行平台
- 具备工作流编排能力
- 具备 Provider 抽象能力
- 具备 Runtime 抽象能力
- 具备审计与策略控制能力
- 具备项目洞察与管理能力
- 同时覆盖桌面端、移动端、后台端

### 1.3 实施原则

始终优先：

- 小而清晰、可测试的迭代切片
- 稳定的抽象边界
- 可见的任务流转
- 强回归保护

尽量避免：

- 大块未测试重写
- 隐式行为变化
- 没有版本边界的功能堆叠

## 2. 质量门禁

这些规则适用于每个版本。

### 2.1 测试是必需项

任何有意义的功能变更，都必须补测试或更新测试。

最低要求：

- 领域逻辑单元测试
- 服务/API 集成测试
- 重要缺陷的回归测试

### 2.2 回归规则

只要修了 bug：

- 必须同步补回归测试
- 不能只改代码不补保护

### 2.3 安全规则

涉及以下边界的功能必须考虑安全测试：

- 认证
- 权限
- 项目可见性
- 危险操作
- 回滚
- 外部 Provider 密钥
- 后台管理操作
- AI 洞察配置

### 2.4 性能规则

涉及以下热点时必须做性能验证：

- 任务看板加载
- 队列操作
- Agent 事件流
- 审计入库
- 移动端摘要 API
- 后台仪表盘
- AI 洞察生成链路

### 2.5 版本完成条件

一个版本不满足以下全部条件，就不能算完成：

- 核心范围已实现
- 快乐路径有测试覆盖
- 关键失败路径有测试覆盖
- 已知问题有记录
- 本版本修复的 bug 有回归测试
- 涉及的安全边界经过检查
- 涉及的热点路径做过性能检查

## 3. 测试分层

### 3.1 单元测试

适用场景：

- 任务状态流转
- 工作流迁移判断
- Provider 路由
- 洞察场景路由
- 策略判定
- 快照和回滚规则

### 3.2 集成测试

适用场景：

- HTTP API
- WebSocket 事件
- 任务分配行为
- 认领和执行流程
- 审批与验收流程
- 持久化行为

### 3.3 端到端与冒烟测试

适用场景：

- 看板能正常加载
- 任务能创建并认领
- Agent 执行后任务状态会变化
- 文档播种任务能出现
- 后台摘要页能正常返回
- 移动端摘要接口能正常返回

### 3.4 回归测试

以下问题必须补回归：

- 每个 P1 或 P2 bug
- 每个线上 bug
- 每个并发 bug
- 每个状态机 bug
- 每个权限 bug

### 3.5 安全测试

以下能力必须补安全检查：

- 角色和能力边界
- 项目隔离
- 危险操作记录
- 回滚授权
- Provider 密钥处理
- 私有化部署配置

### 3.6 性能测试

以下路径必须有性能基线：

- 看板列表接口
- 队列操作
- 大审计查询
- 事件入库
- AI 洞察缓存命中和未命中

## 4. 版本规划

Spotlight 至少要按 10 个以上版本渐进构建。

从现在开始统一采用新的版本规则：

- `0.1.0`
- `0.1.1`
- `0.1.2`
- ...
- `0.1.10`
- `0.2.0`
- `0.2.1`
- 后续继续按语义化递进

说明：

- 对外展示可以简写成 `0.2`
- 代码、配置、安装包、Cargo、npm 等需要合法版本号时，一律使用完整语义化格式，例如 `0.2.0`

下面这份路线图定义了第一阶段的 `0.1.0` 到 `0.1.10`。

## 7. 协作协议

执行任务时：

1. 先确保任务已经存在于看板
2. 明确认领任务
3. 范围要对齐当前版本计划
4. 架构或产品行为变化时同步更新文档
5. 标记任务完成前先补测试

创建新任务时：

- 从版本计划里拆
- 从实现中发现的缺口里拆
- 从测试失败、bug 报告、文档不一致里拆

## 8. 缺陷分级

所有重要 bug 都必须分级：

- `P0`
  - 数据丢失、安全泄漏、不可逆损坏、认证边界失效
- `P1`
  - 核心流程不可用、严重状态损坏、重大协作阻塞
- `P2`
  - 重要功能不正确，但存在绕过方式
- `P3`
  - 轻微体验问题或低风险正确性问题

最低回归规则：

- `P0`、`P1`、`P2` 一律必须补回归测试

## 9. 安全检查清单

任何触及信任边界的任务，在关闭前都要确认：

- 权限检查正确
- 项目隔离正确
- Provider 密钥处理安全
- 审计事件正确写入
- 没有危险的后台绕过
- 回滚和危险操作的可见性被保留

## 10. 性能检查清单

任何触及热点路径的任务，在关闭前都要确认：

- 延迟有测量
- 返回体体积合理
- 缓存行为清楚
- 没有明显的 N+1 或重复计算
- 已标出未来可以补 benchmark 或回归基线的入口

## 11. 完成定义

一个任务不满足以下条件，就不能标记完成：

- 实现已经存在
- 测试已经补齐或更新
- 文档已经对齐
- 已考虑回归风险
- 已考虑安全影响
- 已考虑性能影响

当版本规划、质量门禁、语言规则发生重要变化时，必须同步更新本文件。
