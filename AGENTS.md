# AGENTS.md

本文件是 `C:\Users\zuoho\code\spotlight` 内所有人类协作者、Agent、以及不同会话共享的执行手册。

Spotlight 的目标不只是“把产品做出来”。
更重要的是：用 Spotlight 自己的任务协作方式，把 Spotlight 一步一步自举出来。

## 0. 语言规则

本仓库从现在开始统一采用中文为主。

硬性规则：

- 对话沟通默认使用中文

## 1. 工作原则

### 1.1 唯一事实来源

产品和架构决策的事实来源是：

- `docs/`
- 尤其是：
  - `docs/product-constraints-v1.md`
  - `docs/system-architecture.md`
  - `docs/data-model-v2.md`
  - `docs/platform-abstractions-v1.md`
  - `docs/admin-and-ops-console.md`
  - `docs/ai-management-and-insight-engine.md`
  - `docs/agent-autonomy-and-decision-rules.md`

如果代码和文档冲突：

- 不能悄悄忽略
- 要么把代码对齐文档
- 要么在同一次修改里同步更新文档

### 1.2 产品定位

Spotlight 不是：

- 某一个 CLI 的壳子
- 只有任务列表和聊天框的工具

Spotlight 要成长为：

- 多人协作的 AI 执行平台
- 具备工作流编排能力
- 具备 Provider 抽象能力
- 具备 Runtime 抽象能力
- 具备审计与策略控制能力
- 具备项目洞察与管理能力
- 同时覆盖桌面端、移动端、后台端

### 1.3 实施原则

始终优先：

- 小而清晰、可测试的迭代切片
- 稳定的抽象边界
- 可见的任务流转
- 强回归保护

尽量避免：

- 大块未测试重写
- 隐式行为变化
- 没有版本边界的功能堆叠

## 2. 质量门禁

这些规则适用于每个版本。

### 2.1 测试是必需项

任何有意义的功能变更，都必须补测试或更新测试。

最低要求：

- 领域逻辑单元测试
- 服务/API 集成测试
- 重要缺陷的回归测试

### 2.2 回归规则

只要修了 bug：

- 必须同步补回归测试
- 不能只改代码不补保护

### 2.3 安全规则

涉及以下边界的功能必须考虑安全测试：

- 认证
- 权限
- 项目可见性
- 危险操作
- 回滚
- 外部 Provider 密钥
- 后台管理操作
- AI 洞察配置

### 2.4 性能规则

涉及以下热点时必须做性能验证：

- 任务看板加载
- 队列操作
- Agent 事件流
- 审计入库
- 移动端摘要 API
- 后台仪表盘
- AI 洞察生成链路

### 2.5 版本完成条件

一个版本不满足以下全部条件，就不能算完成：

- 核心范围已实现
- 快乐路径有测试覆盖
- 关键失败路径有测试覆盖
- 已知问题有记录
- 本版本修复的 bug 有回归测试
- 涉及的安全边界经过检查
- 涉及的热点路径做过性能检查

- `0.1.0`
- `0.1.1`
- `0.1.2`
- ...
- `0.1.10`
- `0.2.0`
- `0.2.1`

- 对外展示可以简写成 `0.2`

## 7. 协作协议

执行任务时：

1. 先确保任务已经存在于看板
2. 明确认领任务
3. 范围要对齐当前版本计划
4. 架构或产品行为变化时同步更新文档
5. 标记任务完成前先补测试

自动认领下一个任务时：

1. 必须先全局扫描当前任务列表，不能只看局部上下文
2. 如果存在明确优先级，必须优先选择高优任务，再考虑中优和低优
3. 如果没有明确优先级，Agent 必须先梳理任务之间的前后依赖、阻塞关系和实现顺序，再选择逻辑上最应该先做的那个任务
4. 如果某个任务是其他任务的前置条件，必须优先完成前置任务，不能跳过依赖直接做后置任务
5. 任一 Agent 在任一时刻只能主动执行一个任务，禁止同时启动多个任务、并行认领多个任务、或在一个任务未收口前切换到另一个任务
6. 只有当前任务完成、明确阻塞、或被人工终止后，才允许重新评估并认领下一个任务
7. 当 Agent 选择某个任务作为下一个任务时，应在任务记录或活动日志里说明选择依据，至少说明是基于优先级还是基于依赖顺序

Agent 自治与决策边界：

1. 详细规则以 `docs/agent-autonomy-and-decision-rules.md` 为准，所有 Agent 都必须遵守
2. Agent 默认不是被动工具，而是要主动发现系统问题、流程缺口、功能闭环缺口和界面冗余
3. Agent 可以自行决定既有目标和架构边界内的小到中等实现决策，不要对低风险事项反复等待人工确认
4. Agent 必须把长期有效的规则、架构结论、取消范围和重复故障沉淀到文档或任务记录，不能只留在一次性会话里
5. 低频优化、非主路径体验改进、长期技术债和可选增强，如果当前不是主阻塞，应进入等待队列，不要打断主线
6. 改变产品目标、安全边界、权限边界、计费模型、租户边界或进行破坏性迁移时，必须请示
7. 最新明确决策优先于旧描述；如果任务活动、项目聊天室或用户指令里已经出现“取消、撤销、先不做、去掉、删除、放弃”等信号，Agent 必须立即收缩范围，禁止继续实现对应子需求
8. 每完成一个任务后，必须重新全局评估是否产生后续任务、是否需要重排优先级、是否有新规则需要沉淀

创建新任务时：

- 从版本计划里拆
- 从实现中发现的缺口里拆
- 从测试失败、bug 报告、文档不一致里拆

## 8. 缺陷分级

所有重要 bug 都必须分级：

- `P0`
  - 数据丢失、安全泄漏、不可逆损坏、认证边界失效
- `P1`
  - 核心流程不可用、严重状态损坏、重大协作阻塞
- `P2`
  - 重要功能不正确，但存在绕过方式
- `P3`
  - 轻微体验问题或低风险正确性问题

最低回归规则：

- `P0`、`P1`、`P2` 一律必须补回归测试

## 9. 安全检查清单

任何触及信任边界的任务，在关闭前都要确认：

- 权限检查正确
- 项目隔离正确
- Provider 密钥处理安全
- 审计事件正确写入
- 没有危险的后台绕过
- 回滚和危险操作的可见性被保留

## 10. 性能检查清单

任何触及热点路径的任务，在关闭前都要确认：

- 延迟有测量
- 返回体体积合理
- 缓存行为清楚
- 没有明显的 N+1 或重复计算
- 已标出未来可以补 benchmark 或回归基线的入口

## 11. 完成定义

一个任务不满足以下条件，就不能标记完成：

- 实现已经存在
- 测试已经补齐或更新
- 文档已经对齐
- 已考虑回归风险
- 已考虑安全影响
- 已考虑性能影响

当版本规划、质量门禁、语言规则发生重要变化时，必须同步更新本文件。

## 12. 解决问题总规则：先找参考，再为我所用

每次面对架构决策、新功能设计、或技术难题时，Agent 必须遵守以下步骤：

### 12.1 先搜索，不要从零开始

在动手实现之前，先在 GitHub 和技术社区搜索是否已有成熟的开源方案或设计模式。
搜索目标包括但不限于：

- 类似功能的开源项目
- 相关架构设计文档
- 已发表的论文或技术博客
- 行业最佳实践

### 12.2 评估与借鉴标准

找到参考后，按以下维度评估是否值得借鉴：

| 维度 | 问题 |
|------|------|
| 问题匹配度 | 它解决的问题和我们的问题有多大重叠？ |
| 架构兼容性 | 它的设计能否融入 Spotlight 现有架构？ |
| 成熟度 | 它是实验项目还是经过生产验证的？ |
| 可提取性 | 能否只提取核心思路而不引入整个依赖？ |
| 许可证 | 是否允许借鉴？ |

### 12.3 借鉴方式

不是复制粘贴代码，而是：

1. 提取核心设计思路和模式
2. 用 Spotlight 的架构语言重新实现
3. 在文档或任务记录中注明参考来源
4. 保留参考项目的链接供后续追踪

### 12.4 当前重点参考项目

以下项目与 Spotlight 的目标高度相关，Agent 在相关领域做决策时应优先参考：

#### 多 Agent 并行编排

- [ComposioHQ/agent-orchestrator](https://github.com/ComposioHQ/agent-orchestrator)
  - 核心价值：每个 Agent 独立 worktree + 独立分支 + 独立 PR；CI 失败自动修复；编排器本身也是 Agent
  - 可借鉴：执行槽隔离模型、CI 反应机制、notification-driven 的人机协作模式
